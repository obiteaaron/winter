## 项目内容

Winter项目作为过去开发者的一个项目经验，以及作为学习练手的项目，提供了以下三个核心的功能模块：
1. 一开始做的是配置中心，因为我之前阅读过Diamond（开源的是Nacos）的源码，所以比较了解实现方式，就在我们的单体应用中做了一个简单的版本。当时的主要原因是，独立部署客户无法部署C/S架构的配置中心，只能自己做一个单机版本的。
2. 后来又因为同样的原因，参考了一些PowerJob的源码，实现了一个单机、单集群的调度中心。说是调度中心，其实就是一个调度工具包，并没有很强大的调度管理能力，只是方便了业务在集群内调度的工具包。
3. 再后来是在一个云上、弹内系统需要打通的业务场景中，实现了一个RPC框架，这部分也是因为阅读过Dubbo的源码，同时因为网络不通的原因，直接写HTTP接口太多，所以参考Dubbo的实现做了一个更易用的、基于HTTP的RPC框架调用。这个的实现还是比较完整的。

以上三个模块都是生产验证过的，虽然比较简陋，但我个人认为还是比较可靠的实现。

## 项目现状

这个项目的产生背景是2023年裁员之际，我当时工作也有一些不顺，所以比较焦虑，就想着做点开源项目来缓和一下，同时也相当于把一些没有在工作中做好的事情，重新做一遍，但重新做一遍，也因为没有应用场景，也并没有做很好。

到目前位置，这个项目暂时停止更新，原因有三：
1. 因为过去一段时间的项目原因所产生的这样一个系统，所以设计的是一个单体应用的解决方案。这不符合当前社会上的技术方案路径（当前都是微服务化的），也没有很强的技术含量，因此提供也给技术Demo可用，也给需要的人一些启示即可。
2. 经过我的学习，我认为 gRPC 确实做的很好，应用也很广泛，我认为当时我自研了一个中间件而不是使用 gRPC 是一个不太好的决定，虽然我做的中间件也生产可用，而且稳定性还不错，但只是一个简单的 HTTP 封装，无法推广给更多项目使用。这算是一个比较不错的学习项目了，但和 gRPC 这种相去甚远，也失去了一次在生产使用 gRPC 的机会（肯定会有坑，比如网关不支持HTTP/2需要怎么解决）。
3. AI Coding 从 2025 年初开始发展的很火爆，到今天，基本可以确定，AI Coding代替程序员的时代真的马上到了。这种情况下，我提供给大家一个思路，AI 顺着这个思路立刻就能实现一个生产可用的嵌入式中间件了，甚至可能比我写的代码更好。
